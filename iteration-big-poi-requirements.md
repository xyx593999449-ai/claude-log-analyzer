# 大 POI 可视化改造需求文档

## 1. 背景

当前项目已经具备通用日志上传与分析能力，但现阶段的核心目标不再只是查看原始 Claude Code 日志，而是围绕“大 POI 核实技能”的执行情况，提供面向业务任务的可视化页面，用于展示核实结果、质检结果及其执行过程。

## 2. 迭代目标

本次迭代需要在现有项目基础上新增一个面向业务任务的首页，用于对核实与质检结果进行可视化展示，并支持在需要排查时切换到日志详情页查看任务关联日志。

## 3. 需求范围

### 3.1 概要展示

页面需要提供 4 个概要模块。

#### 模块 1：所有任务

数据来源：`poi_init`

需要展示：

- 总任务量
- 各 `verify_status` 状态数量

#### 模块 2：已核实任务

数据来源：核实日志分析结果 + 临时分析表

需要展示：

- 核实任务量
- 整体核实总用时
- 平均单条核实用时
- 平均单条输入 Token
- 平均单条输出 Token
- 平均单条成本

#### 模块 3：已质检任务

数据来源：质检日志分析结果 + 临时分析表

需要展示：

- 质检任务量
- 整体质检总用时
- 平均单条质检用时
- 平均单条输入 Token
- 平均单条输出 Token
- 平均单条成本

说明：质检执行概览的口径与核实执行概览保持一致，主状态字段以 `qc_status` 为准。

#### 模块 4：人工任务监控

数据来源：`poi_verified`、`poi_qc`、导入记录、临时分析表

需要展示：

- 当前需人工核实任务数量
- 当前需人工核实任务列表摘要
- 最近一次日志导入时间与规模
- 关联异常任务数量

本模块的完整业务流程仍在建设中，因此本次迭代先预留模块位，并先展示“最终需要人工核实”的任务。

人工任务判定规则：

- `poi_qc.is_manual_required = 1`
- 或 `poi_verified.verify_status = "需人工核实"`

### 3.2 详情展示

详情展示按任务维度组织，每条任务一行记录，主关联键默认使用 `task_id`。页面默认分页，并支持分页条数设置。

每条任务需要展示以下内容：

#### POI 基本信息

来源：`poi_init`

字段范围至少包括：

- `task_id`
- `id`
- `name`
- `city`
- `address`
- `poi_type`
- `verify_priority`
- `verify_status`

#### 执行状态

来源：日志分析结果与业务表组合

需要展示：

- 当前核实执行状态
- 当前质检执行状态
- 是否已落库
- 是否存在异常 / 关联缺失
- 是否存在“日志结果与数据库结果不一致”

执行状态口径要求：

- 核实状态以数据库中的实际结果为准
- 质检主状态字段以 `qc_status` 为准
- 如果日志分析结果与数据库结果不一致，需要用警示图标和悬浮说明标记异常原因

#### 核实后成果

来源：`poi_verified`

需要展示：

- `verify_status`
- `verify_result`
- `overall_confidence`
- `verify_time`
- `verification_notes`
- `verify_info`
- `evidence_record`

#### 核实详情

来源：核实日志分析结果

需要展示：

- 最终执行结论
- 开始时间
- 结束时间
- 用时
- 重试次数
- Session 数
- 输入 Token
- 输出 Token
- 成本
- 错误摘要

#### 质检情况

来源：`poi_qc` + 质检日志分析结果

需要展示：

- `qc_status`
- `quality_status`
- `verify_result`
- `qc_score`
- `is_qualified`
- `has_risk`
- `qc_time`
- `qc_result`
- 质检执行详情

详情展示策略：

- 默认提炼关键字段展示
- 长文本、长 JSON、证据数组和结构化详情支持展开查看

### 3.3 交互要求

- 新页面作为首页
- 原有通用日志分析页保留，作为日志详情查看页使用
- 首页提供两个独立上传窗口：
  - 核实日志上传
  - 质检日志上传
- 日志导入默认追加
- 页面提供“清除缓存”按钮
- 页面风格保持与原项目一致
- 长内容支持展开 / 收起
- 过长 JSON 采用折叠区域或小号等宽字体展示
- 支持搜索与筛选
- 列表默认展示概要信息，展开后展示深层详情

清除缓存边界：

- 只清除日志解析落表结果和导入记录
- 不允许修改、删除或清空 `poi_init`、`poi_verified`、`poi_qc`

推荐筛选项：

- 核实状态
- 质检状态
- 城市
- 是否需人工核实
- 是否存在异常

## 4. 数据来源与输入资料

### 日志样例

- 核实过程日志：`example/verify_log`
- 质检过程日志：`example/qc_log`

日志解析要求：

- 支持流式日志场景
- 优先使用 `task_id` 进行关联
- 对个别缺失 `task_id` 的日志，允许基于前后日志段落和上下文做补关联
- 每个任务执行开始和结束存在明显提示词，需要在实现阶段设计任务段落识别规则

### 数据库资料

- 连接配置：`example/db_conf/db_config.yaml`
- 表结构：`example/db_conf/tables_ddl.sql`
- 样例数据：`example/db_conf/sample_data.json`

## 5. 约束条件

- 正式数据库当前不可连，开发阶段必须使用 mock 数据库
- 不修改现有业务表结构
- `poi_init`、`poi_verified`、`poi_qc` 只读，禁止做任何修改和删除动作
- 临时分析结果需要单独落表
- 页面查询必须通过数据库读数，不能每次打开页面都重新全量解析大日志
- 正式库目标为 PostgreSQL，开发时需要保留可替换的数据访问层

## 6. 本轮默认假设

- 主关联键使用 `task_id`
- mock 数据库默认采用 SQLite
- 新增临时分析表默认命名为 `temp_task_analysis`
- 页面以“首页看板 + 列表行展开详情 + 日志详情页切换”方式实现
- 日志导入默认追加
- 清缓存只清临时分析结果

## 7. 补充需求

- 需要展示导入快照信息，包括导入时间、导入任务量、关联失败量
- 需要对异常任务做显式标记
- 列表中建议提供“查看日志”入口，便于切换到日志详情页追溯
- 搜索建议覆盖 `task_id`、POI ID、名称、地址、城市
- 详情中原始大字段采用折叠展示

## 8. 非目标

- 不修改核实算法本身
- 不修改质检判定规则
- 不在本轮接入正式数据库
- 不在本轮处理部署方案
