# 大 POI 可视化二期（效果优化）开发计划

## 1. 目标
- 在不回退现有功能的前提下，恢复原页面能力并完成交互/视觉优化。

## 2. 开发拆解
### 阶段 A：页面结构与导航恢复
- 增加顶层路由状态：`dashboard` / `legacy-analysis` / `task-log-detail`。
- 恢复原始日志分析页面入口（优先复用原组件，缺失部分补回）。
- 保证默认首页仍是新看板。

### 阶段 B：上传交互升级
- 上传组件改为可复用 `UploadDropzone`：
  - 支持拖拽
  - 支持批量选择
  - 支持多文件列表与删除
- 增加识别器 `logClassifier`：
  - 输入：文件名 + 前 N 行内容
  - 输出：`verify|qc` + `executor|claude` + 置信度
- 导入前展示识别结果，可手动调整。

### 阶段 C：成本与 Token 口径统一
- 后端增加统一成本计算函数（GLM 定价常量集中定义）。
- 概览与详情接口统一返回：
  - `totalTokens`
  - `inputTokens`
  - `outputTokens`
  - `inputCost`
  - `outputCost`
  - `totalCost`
- 前端 Tooltip 统一显示公式与单价说明。

### 阶段 D：详情视觉重构
- 详情表格升级为“分组卡片行”或“层级化表格”：
  - 主信息行 + 展开子面板
  - 状态与风险标签规范化
  - 文本折叠统一规则
- 保留分页和筛选行为不变。

## 3. 交付清单
- 新增/改造上传组件与分类器。
- App 导航结构恢复并可切换原页面。
- 成本与 Token 统一口径实现（含 tooltip）。
- 详情区样式与布局重构。
- 本次迭代文档（需求/计划/验收）更新完成。

## 4. 实施顺序
1. 导航与页面恢复
2. 上传交互优化 + 自动识别
3. 成本与 Token 口径统一
4. 详情视觉重构
5. 联调与验收

## 5. 风险控制
- 先保留旧交互入口，避免一次性替换导致回归范围过大。
- 上传识别失败时必须支持人工纠正，避免阻塞导入流程。
- 成本口径切换后同步核对看板与详情数值一致性。

## 6. 本轮追加实施项（日志详情复用）
- 在“阶段 A”后新增 `阶段 A2`：
  - 恢复并复用原项目日志详情组件（时间线/工具分析/Token/原始日志）。
  - 将 `TaskLogPage` 从简版文本展示替换为原日志详情视图容器。
  - 适配任务维度数据源（按 task_id 关联的 session 集合），保证原组件可直接消费。
- 交付要求：
  - 点击“日志详情”进入后，视觉与交互接近原页面；
  - 返回首页与筛选状态不丢失；
  - 不影响当前看板查询与分页。
